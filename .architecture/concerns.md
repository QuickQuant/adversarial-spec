## Top Concerns

### CON-001: Triple litellm completion() pathway creates silent behavioral drift
- **Priority:** next
- **Severity:** warning
- **Component:** model-dispatch
- **Summary:** Three files independently call litellm.completion() with different defaults (temperature 0.3 vs 0.7, max_tokens 4000 vs 4096 vs 8000). A change to retry logic or error handling in one path won't propagate. The behavioral drift is undocumented and brittle.
- **Recommended action:** Create a single low-level `_call_litellm()` in model_dispatch.py that all three sites delegate to, with temperature/max_tokens as explicit parameters.
- **Files:** models.py, gauntlet/model_dispatch.py, debate.py
- **Source refs:** FIND-001, FIND-014
- **Confidence:** high

### CON-002: cost_tracker coupling forces 30+ test monkeypatches
- **Priority:** next
- **Severity:** warning
- **Component:** gauntlet-pipeline
- **Summary:** Every gauntlet phase imports the global cost_tracker singleton from models.py. This is a layer violation (gauntlet reaching into debate-engine internals) and creates a testing burden (30+ monkeypatch lines across test files). The gauntlet's model_dispatch.call_model() already sits at the right boundary.
- **Recommended action:** Move cost_tracker.add() into model_dispatch.call_model() so phase files never touch cost tracking directly. Tests would need only one monkeypatch point.
- **Files:** gauntlet/model_dispatch.py, models.py, all phase_*.py files
- **Source refs:** FIND-006
- **Confidence:** high

### CON-003: orchestrator.py run_gauntlet() is a 695-line function
- **Priority:** next
- **Severity:** warning
- **Component:** gauntlet-pipeline
- **Summary:** The function that sequences all gauntlet phases is ~695 lines with 30+ local variables and 3-level try/except nesting. Mixes phase orchestration with checkpoint, manifest, medal, and stats logic. Any phase sequencing change requires understanding the entire function.
- **Recommended action:** Extract a phase-table pattern: each phase gets a (name, entry_function, checkpoint_key) tuple, and the orchestrator loops over the table with shared checkpoint/resume logic.
- **Files:** gauntlet/orchestrator.py
- **Source refs:** FIND-011
- **Confidence:** high
